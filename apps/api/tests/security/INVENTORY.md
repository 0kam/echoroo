## Maintenance notes

- When adding new `test_*.py` files under `tests/security/**`, increment the
  category total in this document and re-confirm 75+ aggregate.
- CI gate (PR-007): `pytest tests/security --co -q | wc -l` must remain ≥ 75.
- The legacy `/api/v1/*` Bearer-auth tests in
  `tests/security/authorization/test_member_vs_admin.py`,
  `test_viewer_permission_boundary.py`,
  `test_viewer_precise_location_denied.py`,
  `tests/integration/test_permissions.py`, etc. observe **RBAC matrix coverage**
  (Owner/Admin/Member/Viewer × action). They do **not** exercise the production
  `/api/v1` API-key surface — instead, the `client` fixture in
  `tests/conftest.py` installs a test-only shim that converts a Bearer JWT into
  a synthetic `Principal.for_api_key(scopes=ALL Permission, project_id=None)`
  via direct `AuthRouterMiddleware._authenticate_api_key` attribute assignment
  (Phase 16 Batch 6c). This means:
  - Production behaviour ("Bearer JWT on `/api/v1` returns 401") is pinned by
    `tests/integration/middleware/test_two_factor_enforcement_real_chain.py`,
    NOT by the RBAC matrix tests above.
  - `_api_key_scopes=ALL` is intentional — scope intersection becomes a no-op
    so the matrix tests can observe pure role-based deny/allow boundaries.
  - **API-key scope-boundary tests** (e.g. an API key restricted to
    `view_detection` denied an `edit_project` request) MUST use real
    `Principal.for_api_key` fixtures — the shim is unsuitable. See the
    Phase 15 `test_superuser_api_key_forbidden.py` and
    `test_permissions_phase15_r3.py` suites for that pattern.
  - Step -1 universal api-key veto for `is_superuser_only` actions (Phase 15
    Batch 4 R4) DOES fire under the shim — superuser-only positive tests
    written against this `client` fixture + Bearer JWT will see a false 403.
- KMS / moto fixtures in `tests/unit/core/test_kms.py` clear
  `AWS_ENDPOINT_URL_KMS` / `AWS_ENDPOINT_URL` before entering `mock_aws()` so
  the dev-container LocalStack endpoint does not bleed into unit tests.
