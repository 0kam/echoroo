# Contributing to Echoroo

Thanks for your interest in contributing to Echoroo! This guide covers how to
set up a development environment, the quality gates your change must pass, and
our pull-request conventions.

By participating in this project you agree to abide by our
[Code of Conduct](CODE_OF_CONDUCT.md).

## Project layout

```
apps/web/   # SvelteKit 2 / Svelte 5 frontend (TypeScript)
apps/api/   # FastAPI backend (Python 3.11) + Celery ML workers
specs/      # Feature specifications
```

## Development setup

The recommended way to run the full stack (Postgres + pgvector, Redis,
LocalStack, backend, frontend, Celery workers) is via Docker:

```bash
cp .env.example .env          # then edit the required values (see README)
./echoroo.sh install          # prepares prerequisites; creates .env if missing
./echoroo.sh checkenv         # validates required .env settings
./echoroo.sh start            # start the development stack
```

The frontend is then served at http://localhost:5173.
`./scripts/docker.sh dev` is a compatibility wrapper that delegates to
`./echoroo.sh`.

Useful lifecycle commands:

```bash
./echoroo.sh status           # containers + health
./echoroo.sh logs             # tail logs
./echoroo.sh migrate          # run Alembic migrations
./echoroo.sh stop             # stop containers, keep data
```

See [DOCKER.md](DOCKER.md) and [CONFIGURATION.md](CONFIGURATION.md) for details,
and the README "ML Configuration" section for GPU/CPU inference settings.

### Running without Docker

You need Python 3.11+, [uv](https://docs.astral.sh/uv/), Node.js 20+, and a
running PostgreSQL instance.

```bash
# Backend (Terminal 1)
cd apps/api && uv run uvicorn echoroo.main:app --reload

# Frontend (Terminal 2)
cd apps/web && npm install && npm run dev
```

## Quality gates

Please run the relevant checks locally before opening a PR. These mirror the
gates enforced in CI (`.github/workflows/ci.yml`).

### Frontend (`apps/web`)

```bash
npm run messages     # compile Paraglide i18n messages (generated, gitignored)
npm run check        # svelte-check — zero type errors required
npm run lint         # eslint — zero errors required (warnings allowed)
npm run test -- --run  # vitest unit suite
```

> Note: `$lib/paraglide/*` message modules are generated by the Vite plugin and
> are gitignored. `svelte-check` / `vitest` do not run that plugin, so run
> `npm run messages` first or you will see "Cannot find module '$lib/paraglide/...'"
> errors.

### Backend (`apps/api`)

```bash
uv sync --all-extras --dev   # install runtime + dev dependencies
uv run ruff check .          # lint
uv run mypy .                # type check
uv run pytest                # test suite
```

## Continuous integration

CI (`.github/workflows/ci.yml`) runs on every push/PR to `main`. The required,
blocking checks include:

- **Frontend type-check + lint** (`svelte-check`, `eslint`) on Node 22.
- **Frontend unit tests** (`vitest`).
- **Frontend paraglide hardcoded-string lint** — new UI text must go through
  Paraglide `m.*()` calls, not hardcoded English.
- **backend-tests** — full pytest suite with a per-module coverage gate
  (permission-critical modules ≥ 95%, others ≥ 85%), run in parallel via
  `pytest-xdist`.
- **security-tests** — the `-m security` pytest suite (always blocking).
- **Permissions strict gate** — permission-guard / response-filter / search-gate
  / no-raw-coordinates / KMS-isolation / hardcoded-license lints plus an OpenAPI
  raw-coordinate leak assertion and an endpoint-coverage reflection test.
- **Supply-chain gate** — `uv sync --locked` wheel-hash verification plus
  `pip-audit` CVE advisory scan.

Additional workflows: `e2e.yml` (opt-in full-stack Playwright), `build_frontend.yml`,
`publish_docker.yml`, and `mutation-testing` (opt-in, ≥ 80% per module).
Mutation and E2E jobs are opt-in via a PR label or `workflow_dispatch`.

## Pull-request conventions

- **Commit messages** follow [Conventional Commits](https://www.conventionalcommits.org/):
  `type(scope): summary`, e.g. `feat(media): ...`, `fix(api): ...`,
  `refactor(search): ...`, `ci: ...`, `docs(api): ...`.
- **Branch names** are typically prefixed by intent or spec number, e.g.
  `009-browser-api-bff-migration`, `experimental/toritore`, `fix/...`. Feature
  work driven by a spec under `specs/` uses that spec's numeric prefix.
- Keep PRs focused; reference the related spec or issue.
- Ensure all applicable quality gates above pass before requesting review.
- All code, comments, and documentation must be written in English.
