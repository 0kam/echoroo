エグゼクティブ・サマリ

結論: メール確認機能の backend / frontend のコード は 実装済みで動作可能だが、dev / preview 環境では実際にメールが届かない ため「機能していない」状態になっている。原因は RESEND_API_KEY が空のまま、 かつ MailHog/Mailpit などの dev SMTP キャッチャが存在しない こと。 エンドユーザーから見た発火点は CLI superuser の未認証バッジ だが、根本は 「メール送信そのものが skip されている」「確認リンクの導線を ユーザーが受け取れない」というインフラ欠落である。

1. 機能はどこまで動くか

1.1 トークン発行〜検証 (コード上は完成)

  1. POST /web-api/v1/auth/register で EmailVerificationService.issue_verification_token() が呼ばれる
  2. DB に email_verification_tokens 行が作られ、AES-GCM で seal した token を発行
  3. 本来はここで Resend API 経由でメール送信 — ただし dev では skip (詳細 §2)
  4. ユーザーがメール本文の /verify-email?token=... をクリック
  5. Frontend が POST /web-api/v1/auth/verify-email 呼出
  6. Service が unseal + users.email_verified_at = now() セット
  7. Profile タブで「メール認証済み」バッジ表示

1.2 Enforcement middleware は default OFF

# apps/api/echoroo/middleware/email_verification_enforcement.py:68-69
if not settings.EMAIL_VERIFICATION_ENFORCEMENT_ENABLED:
    return await call_next(request)  # スキップ (デフォルト)

EMAIL_VERIFICATION_ENFORCEMENT_ENABLED: bool = False がデフォルトのため、 未認証ユーザーでも全エンドポイントを叩ける。 「未認証バッジが出ても実害がないので誰も気付かない」 という構造になっている。

2. 「メール確認機能が機能していない」の正体

核心の発見: dev 環境では Resend API キーが未設定で、メール送信処理が silently skip される。register したユーザーには そもそも確認メールが 届かない。

2.1 Resend SDK が常に「skip モード」

# apps/api/echoroo/services/email.py:110-116
if not settings.RESEND_API_KEY:
    logger.warning(
        "verification email skipped — RESEND_API_KEY not configured "
        "(recipient_hash=%s)",
        recipient_hash,
    )
    return  # ← raise しないので呼び出し元は成功扱い

2.2 dev 環境の env 設定

2.3 dev で「メール確認」を試したときの実挙動

  1. ユーザーが /register でサインアップ
  2. backend ログに verification email skipped — RESEND_API_KEY not configured が出る
  3. メールボックスには何も届かない
  4. Profile を見ると「メール未確認」バッジが永久に残る
  5. ただし enforcement middleware が OFF なので、UI 上は全部使える
  6. 「未確認バッジ」だけが浮いている状態 = これがユーザーが見た問題の正体

3. HTTP /setup と CLI init_superuser の差分

3.1 HTTP setup は自動で verified にする

# apps/api/echoroo/services/setup.py:220-231 (PR #88 / commit c4a40253)
user = User(
    email=email,
    ...
    email_verified_at=datetime.now(UTC),  # ← 自動セット
)

3.2 CLI init_superuser はセットしない

# apps/api/echoroo/scripts/init_superuser.py (現状)
user_row = User(
    email=email,
    password_hash=hash_password(password),
    display_name=display_name,
    two_factor_enabled=True,
    two_factor_secret_encrypted=encrypted_secret,
    ...
    # ← email_verified_at の指定なし (None のまま)
)
根本原因の分析: 同じ「operator が自分で typed した email でブートストラップ する」フローなのに、HTTP 側だけ「typed = verified」とみなす設計が入った。 CLI 側は実装漏れ。
どちらが正しいかは哲学次第:
  • HTTP 側に合わせる → CLI も自動 verified、整合性は取れるが 「メール所有確認」のセキュリティ価値はゼロ
  • CLI 側に合わせる → HTTP setup も verification mail 経由にする、 UX は悪化するが筋は通る

4. ユーザーが「未認証」表示を踏むフロー

  1. 運用者が ./scripts/docker.sh dev でセットアップ
  2. CLI init_superuser または HTTP /setup でアカウント作成
  3. CLI 経由 → email_verified_at = NULL のまま
  4. UI でログイン → Profile に「メール未認証」バッジ
  5. resend ボタンを押す → backend は token を発行し直すが、 send_verification_email() は RESEND_API_KEY 空でスキップ
  6. メールは永久に届かず、バッジが消える手段がない

すなわち今ユーザーが踏んでいるのは 二重の問題:

  1. CLI ユーザーが verified にならない (③ の指摘内容)
  2. resend してもメールが来ない (dev インフラ欠落)

6. 推奨修正オプション

Option A — 最小 (15 分)

対象: scripts/init_superuser.py に email_verified_at=datetime.now(UTC) を追加 + テスト 1 本。
解消する問題: ③ の不整合のみ。CLI superuser のバッジが消える。
残る問題: 一般ユーザーが register したときの「メールが届かない」は未解決。
適している場面: launch 前に最低限の整合性だけ取りたい / そもそも一般 register を まだ開放していない場合。

Option B — 中 (1-2 時間) 推奨

A に加えて:
  • compose.dev.yaml に Mailpit container を追加 (port 8025 UI)
  • services/email.py に「RESEND_API_KEY が空かつ APP_ENV=development なら SMTP (Mailpit) にフォールバック」の分岐
  • または、Resend を使わず aiosmtplib で直接送信するパスを dev 用に切る
  • dev でも実際に確認メールが Mailpit に届き、リンクをクリックして verified に到達する end-to-end が動く
解消する問題: ③ + メール本体が届かない問題。
残る問題: Enforcement middleware は依然 OFF (これは launch 戦略次第)。

Option C — 大 (半日〜)

B に加えて:
  • Enforcement middleware を development でも ON で運用してみる
  • HTTP /setup の auto-verify をやめ、setup 完了時にも verify mail を送る統一フロー
  • 本番向け Resend API key を取得し compose.preview.yaml / production env に注入
  • 「未認証ユーザーが詰まる」エラー画面 (ERR_EMAIL_VERIFICATION_REQUIRED) の UX 整備
解消する問題: セキュリティ仕様としてメール確認を完全に機能させる。
適している場面: 一般ユーザーに開く launch 直前。
